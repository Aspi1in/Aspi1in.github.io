Linux/Android 's UID & PID

Posted on 2018-11-13

Linux’s UID、PID

UID（User Identifier）
字面上理解就是用户ID的意思，Linux系统中的用户(UID)分为3类，即普通用户、根用户、系统用户。

普通用户是指所有使用Linux系统的真实用户，这类用户可以使用用户名及密码登录系统。Linux有着极为详细的权限设置，所以一般来说普通用户只能在其家目录、系统临时目录或其他经过授权的目录中操作，以及操作属于该用户的文件。通常普通用户的UID大于500，因为在添加普通用户时，系统默认用户ID从500开始编号。

根用户也就是root用户，它的ID是0，也被称为超级用户，root账户拥有对系统的完全控制权：可以修改、删除任何文件，运行任何命令。所以root用户也是系统里面最具危险性的用户，root用户甚至可以在系统正常运行时删除所有文件系统，造成无法挽回的灾难。所以一般情况下，使用root用户登录系统时需要十分小心。

系统用户是指系统运行时必须有的用户，但并不是指真实的使用者。比如在RedHat或CentOS下运行网站服务时，需要使用系统用户apache来运行httpd进程，而运行MySQL数据库服务时，需要使用系统用户mysql来运行mysqld进程。在RedHat或CentOS下，系统用户的ID范围是1~499。下面给出的示例显示的是目前系统运行的进程，第一列是运行该进程的用户。

PID（Process Identifier）
PID就是各进程的身份标识,程序一运行系统就会自动分配给进程一个独一无二的PID。进程中止后PID被系统回收，可能会被继续分配给新运行的程序。

Android’s UID、PID

Android是一个单用户系统（貌似现在有访客模式了，不知道是不是改了），因为一个非常重要的安全特性，就是沙箱模式，这可以保证不同的应用之间是彼此隔离的，比如对资源的隔离、设备管理的隔离和内存数据的隔离，这样就能很好的爆炸安全问题。实现的主要方法就是系统设计了在当应用安装时被分配了一个UID，不同的应用的UID是不同的（不是绝对的），这样彼此之间就像Linux上登录的不同账户一样，彼此是隔离的，无法互相访问的，这样就实现了沙箱模式。如下图所示：

基本概念

smali和backsmali是dalvik虚拟机使用的dex文件的汇编器和反汇编器，它宽松的语法基于Jasmin/dedexer语法，并支持dex格式的所有功能。

寄存器的数量在函数起始之后通过类似.registers 4的方式约定，还可以通过类似.locals 2的方式确定，两者的区别在于，.registers表示函数使用的所有的寄存器个数，而.locals表示函数使用的非参数占用的寄存器个数，函数使用的寄存器的个数由参数占用的寄存器个数 + 非参数占用的寄存器个数共同构成。

所以在上面的例子中的add()方法中可以看到.registers 4表示两个Integer类型的参数占用了两个寄存器，再加上函数中使用的v0、v1两个寄存器，一共四个寄存器。

参数总是存放在寄存器的后面，如add()中的两个Integer类型的参数就分别存放在v3、v4两个寄存器中，但是在本例中并没有发现在函数中出现v3、v4两个寄存器，这是因为不同的命名方案导致的，p命名方案会将参数改写成Px的形式，p0、p1分别表示了v3、v4，这样会一定程度上便于阅读smali代码，更加重要的意义在于，当你想在函数中添加代码，然后需要额外的寄存器支持时，单纯的修改.registers n中的n是不行的，因为用v命名方式函数中所有的在取参数时候都要往后偏移你要增加的寄存器数量，这会很麻烦，显然p命名方式就不会存在这种问题。

dalvik字节码中，寄存器是32位的，可以存放任意类型的值。64位数据类型可以使用连个寄存器来存放。如long和double

dalvik的字节码有两大类型：primitive types和reference types。Reference types是对象和数组，其他的都是primitive type。

元注解

元注解的定义是：可以注解到注解上的注解，这种特殊的注解是由系统定义好的，功能确定的注解，专门作用于自定义注解上的特殊注解。

@Retention

这个注解是声明注解的生命周期，分三种情况：

  • RetentionPolicy.SOURCE 注解只在源码阶段保留，在编译器进行编译时它将被丢弃忽视。
  • RetentionPolicy.CLASS 注解只被保留到编译进行的时候，它并不会被加载到 JVM 中。
  • RetentionPolicy.RUNTIME 注解可以保留到程序运行的时候，它会被加载进入到 JVM 中，所以在程序运行时可以获取到它们。

@Documented

顾名思义，这个元注解肯定是和文档有关。它的作用是能够将注解中的元素包含到 Javadoc 中去。

@Target

Target 是目标的意思，@Target 指定了注解运用的地方，用来指定注解作用的对象，是类、属性、方法、还是局部变量之类的，有一下一些类别：

  • ElementType.ANNOTATION_TYPE 可以给一个注解进行注解
  • ElementType.CONSTRUCTOR 可以给构造方法进行注解
  • ElementType.FIELD 可以给属性进行注解
  • ElementType.LOCAL_VARIABLE 可以给局部变量进行注解
  • ElementType.METHOD 可以给方法进行注解
  • ElementType.PACKAGE 可以给一个包进行注解
  • ElementType.PARAMETER 可以给一个方法内的参数进行注解
  • ElementType.TYPE 可以给一个类型进行注解，比如类、接口、枚举

@Inherited

这个注解是用来表示可以被继承，意思就是：ClassA是一个被@Inherited注解过的注解@InheritedAnnotation注解的类，那么当ClassB继承了ClassA，ClassB这个类也会拥有@InheritedAnnotation这个注解的信息。

这个元素是用来标记过时的元素，想必大家在日常开发中经常碰到。编译器在编译阶段遇到这个注解时会发出提醒警告，告诉开发者正在调用一个过时的元素比如过时的方法、过时的类、过时的成员变量。

注解的使用

官方的解释是：
注解是一系列元数据，它提供数据用来解释程序代码，但是注解并非是所解释的代码本身的一部分。注解对于代码的运行效果没有直接影响。

注解有许多用处，主要如下：

  • 提供信息给编译器： 编译器可以利用注解来探测错误和警告信息
  • 编译阶段时的处理： 软件工具可以用来利用注解信息来生成代码、Html文档或者做其它相应处理。
  • 运行时的处理： 某些注解可以在程序运行的时候接受代码的提取

所以说注解里面只有属性，没有方法啥的，只是一个标签的意义，告诉系统、编译器或者是其他的类你具有什么特点。

注解主要针对的是编译器和其它工具软件(SoftWare tool)。

当开发者使用了Annotation 修饰了类、方法、Field 等成员之后，这些 Annotation 不会自己生效，必须由开发者提供相应的代码来提取并处理 Annotation 信息。这些处理提取和处理 Annotation 的代码统称为 APT（Annotation Processing Tool)。

最近组里的同学说Shark（一款在Android上抓包的应用）在他的Nexus 5X的Android6.0系统上不能用，主要就是抓包完了之后，pcap文件保存下来的文件没有内容（文件大小是0byte），刚好前几天我妹子说她的Shark不能用了，报了个错，所以我想看看咋回事儿不能用了，要不再整个可以用的，于是就写个这个东西。

tcpdump

tcpdump是Linux系统中一个非常重要的工具，以一个ELF文件的形式存在，不过貌似这个东西开源了，之前在Windows上做过抓包的工具，是用WinPcap驱动提供的接口抓的，WinPcap应该是在网卡驱动上加了个Filter，然后copy一份流量通过接口提供给上层的，tcpdump单单一个ELF文件，我确实不知道他是如何实现数据包捕获的，有空还是要看下代码的。这里我们编写的工具的核心实现就是利用tcpdump来做。

大致思路

没有看过Shark的代码，但是猜测就是用的tcpdump，在他的应用目录下果然看见了tcpdump文件，所以我也是这么做的，没去仔细研究为啥他在高版本系统不能用了，说下我怎么做的，很简单：

  • 释放tcpdump文件到系统中
  • 给tcpdump文件加权限
  • 利用Runtime.getRuntime().exec()方法执行命令启动tcpdump
  • kill掉tcpdump进程，保存生成的pcap包

遇到的问题

流程思路很简单，但是实际做的过程中遇到蛮多问题的：

1.文件无法释放到/data/local/tmp/目录下
抓包应用肯定是要有root权限的，所以可以随意使用命令的方式操作文件，但是tcpdump文件是在编译的时候被放到工程的asset目录下，我们代码里面必须先释放这个文件到一个本地目录下，最开始我选择了释放到/data/local/tmp/目录下，因为网上很多大佬是这么搞得，但是高版本的APP已经没有权限在/data/local/tmp/目录中Create File了，可以new File但是当执行File.CreateNewFile()方法时就会报错，报错说没有权限，其实读写SD卡的权限已经有了，而且/data/local/tmp/目录也不在SD卡中，据说需要SystemApp的级别才能有这个权限，最后干脆调用this.getFilesDir()方法，直接释放到应用自身的目录的中去。

2.关于在Runtime.getRuntime().exec()中执行会阻塞的方法的处理办法
这里存在两个问题：1.抓包的命令执行后，如何获取命令行返回的内容？ 2.如何终止抓包？
最开始直接在UI线程支持命令，命令被阻塞，UI线程直接卡着不动，显然是要开一个线程的，在新的线程中执行Runtime.getRuntime().exec()命令，但是想获取控制台源源不断返回的数据怎么办呢，看下面一个Demo：

这里创建一个线程，执行ping www.baidu.com的命令，这个命令执行后会一直阻塞，知道控制台接收到control + c，才会退出，这个和tcpdump执行的时候一毛一样（其实不一样，后面说），这里通过BufferedReader可以不断的从控制台的输出流中读取内容，当然程序也就会阻塞到这里，我们定义了一个doStop()方法，主要就是销毁因为执行命令生成的process对象，他被销毁了，执行cmd命令的线程自然也会被终止销毁。

main函数中测试一下，看执行结果，还是比较舒服的

但是但是但是，当我把这片逻辑移植到工程中的时候发现，fuck，执行cmd命令的进程确实被关掉了，但是生成的pcap文件大小是0byte，这就说明执行的tcpdump并没有消停啊，这就和ping www.baidu.com不一样了啊，执行top命令一看，果然tcpdump还在跑着呢，找到pid，手动执行kill [pid]，果然文件就有大小了，那咋整，没办法啊，找到一个命令ps -C [command]可以查看指定command的进程信息，稍微处理下，从返回的信息中提取出PID，然后调用Runtime.getRuntime().exec()执行kill [pid]命令，杀之就可以了。这里很疑惑是为啥，当时先猜是不是tcpdump是不是内部开了新的进程去执行命令，然后还设置了什么不随爸爸挂掉儿子也挂掉的东西（我其实不确定是不是有这种机制的），后来发现如果在抓着包，把app杀掉，跑着的tcpdump就也自动停了，我也是见了鬼了，可能是我忽略了什么细节，估计要看下tcpdump的源码才能搞明白吧。

3.一个小问题，就是子线程更新UI主线程中的控件的问题
还是开发做的少了，这种基本操作都没有搞好，不过还好我遇到这个问题之后就想到了貌似之前就看到过用Handler解决这个问题的情况，在UI线程声明一个Handler类型的成员变量，但是要记得加上final关键字，要不然传递实例到子进程中的时候，尼玛这个东西在用的时候会是null。我猜测在UI线程构造一个final关键字的控件的实例，然后传递到子线程，估计也可以更新，没具体试，但是感觉没有用Handler优雅。

代码放到内部的git系统中了，明儿让他们用用，看有没有bug。。。。
